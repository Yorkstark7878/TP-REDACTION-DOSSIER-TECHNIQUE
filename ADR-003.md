# ADR-003 — Le backoff exponentiel avec dead letter queue est la stratégie de retry des événements asynchrones RabbitMQ

**Statut** : Accepté  
**Date** : 2025-05-13  
**Auteurs** : LeoChris8  

Voir §7.2 — NotificationModule

---

## Contexte

SupEvents utilise RabbitMQ pour découpler les services via des événements asynchrones (`ticket.confirmed`, `payment.failed`, `event.cancelled`). Les consommateurs (NotificationModule, TicketService) peuvent échouer à traiter un message pour diverses raisons : indisponibilité d'un service tiers (SendGrid), erreur transitoire de base de données, bug applicatif.

Il faut décider comment gérer ces échecs : retenter le traitement, et si oui, combien de fois et avec quelle politique d'espacement ? Que faire des messages qui échouent définitivement ? Sans politique explicite, RabbitMQ remet les messages en tête de queue indéfiniment (comportement par défaut avec `nack + requeue=true`), bloquant potentiellement le traitement de tous les messages suivants.

---

## Décision

Chaque exchange de production est associé à un dead letter exchange (DLX). En cas d'échec de traitement, le message est `nack`é avec `requeue=false` et republié par RabbitMQ dans la DLX avec un TTL croissant (backoff exponentiel : 2s, 4s, 8s, 16s). Après 4 tentatives infructueuses, le message est routé vers une dead letter queue (`*.dlq`) pour analyse et traitement manuel. Une alerte est déclenchée si la DLQ dépasse 10 messages.

---

## Alternatives envisagées

### Retry immédiat avec requeue=true (comportement RabbitMQ par défaut)

En cas d'échec, le message est remis en tête de queue et retraité immédiatement.

Écarté car : si le service tiers (ex. SendGrid) est en panne, le message sera retraité des centaines de fois par seconde, saturant les logs, consommant les ressources et empêchant le traitement des autres messages en queue (head-of-line blocking). Inacceptable en production.

### Retry à délai fixe

Les messages échoués sont remis en queue après un délai fixe (ex. 30s), sans augmentation progressive.

Écarté car : un délai fixe ne s'adapte pas à la nature de la panne. Une indisponibilité de plusieurs minutes de SendGrid générera quand même un flux constant de retries toutes les 30s. Le backoff exponentiel réduit progressivement la pression sur le service défaillant, lui laissant le temps de récupérer.

### Retry géré entièrement côté applicatif (sans DLX RabbitMQ)

Le consommateur implémente sa propre logique de retry en mémoire (boucle avec sleep).

Écarté car : si le processus consommateur redémarre entre deux tentatives (déploiement, crash), le contexte de retry est perdu. La solution basée sur RabbitMQ (TTL + DLX) est durable : les messages survivent aux redémarrages du consommateur.

---

## Conséquences

### Positives

- Les messages échoués ne bloquent pas le traitement des messages suivants (pas de head-of-line blocking)
- La DLQ offre une visibilité complète sur les échecs persistants et permet le rejeu manuel après correction
- Le backoff exponentiel réduit la pression sur les services défaillants et évite les tempêtes de retry
- La durabilité RabbitMQ garantit qu'aucun message n'est perdu en cas de redémarrage du consommateur

### Négatives

- La configuration DLX/TTL dans RabbitMQ est plus complexe qu'un simple retry en code — nécessite une bonne maîtrise de la topologie des exchanges
- Les messages en DLQ requièrent une intervention manuelle ou un outil de monitoring dédié (ex. interface RabbitMQ Management ou script de rejeu)
- Le délai maximum avant DLQ (2+4+8+16 = 30s) peut être trop court pour des pannes longues — certains messages légitimes finissent en DLQ alors que le service se rétablit

### Dette acceptée

Le mécanisme de rejeu automatique depuis la DLQ (après correction du service défaillant) n'est pas implémenté dans ce TP. En production, un outil ou un script de rejeu sera nécessaire pour vider la DLQ et retraiter les messages en échec. Cette automatisation est reportée à la phase d'exploitation.
