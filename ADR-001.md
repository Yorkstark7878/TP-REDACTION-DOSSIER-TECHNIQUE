# ADR-001 — Les tokens JWT stateless sont utilisés comme mécanisme d'authentification applicatif principal

**Statut** : Accepté  
**Date** : 2025-05-13  
**Auteurs** : LeoChris8  

Voir §7.3 — AuthModule

---

## Contexte

SupEvents doit authentifier ses utilisateurs sur toutes les requêtes API. Le système dispose d'un SSO école (OIDC) qui émet des tokens d'identité, mais ces tokens ne sont pas directement exploitables par les services internes (format, durée de vie, claims insuffisants). Il faut donc choisir un mécanisme d'authentification applicatif propre à SupEvents.

Deux forces s'affrontent : la simplicité opérationnelle (pas de session à gérer côté serveur) et la capacité à révoquer immédiatement une session compromise. La plateforme est une application web classique, sans contrainte forte de haute disponibilité distribuée au lancement. Les utilisateurs sont des étudiants et organisateurs — pas de profil à risque élevé justifiant une révocation ultra-rapide.

---

## Décision

Les tokens JWT signés avec RS256 (clé asymétrique) sont émis par l'AuthModule à l'issue du callback OIDC. La durée de vie est fixée à 15 minutes. Un refresh token opaque (stocké hashé en base PostgreSQL) permet l'émission d'un nouveau JWT sans re-authentification. Les sessions actives sont également trackées dans Redis avec un TTL de 15 minutes pour permettre une invalidation rapide si nécessaire.

---

## Alternatives envisagées

### Sessions serveur via Redis uniquement

Un identifiant de session opaque est stocké dans un cookie ; la session complète réside dans Redis. À chaque requête, le serveur fait un appel Redis pour valider et récupérer les données de session.

Écarté car : introduit une dépendance synchrone sur Redis sur chaque requête authentifiée, ce qui crée un point de défaillance unique. En cas d'indisponibilité Redis, toutes les requêtes authentifiées échouent. Le JWT permet une validation sans appel réseau supplémentaire.

### JWT longue durée (24h) sans refresh token

Un JWT valable 24 heures est émis, sans mécanisme de refresh. Simple à implémenter.

Écarté car : une durée de 24h est inacceptable du point de vue sécurité — en cas de vol du token, l'attaquant dispose d'une fenêtre de 24h sans possibilité de révocation. Le système de refresh avec courte durée de vie offre un compromis bien meilleur.

### Token opaque avec lookup en base à chaque requête

Un token aléatoire est stocké en base. Chaque requête fait un SELECT pour valider le token.

Écarté car : le volume de requêtes engendre une pression importante sur PostgreSQL. Redis avec TTL est plus adapté pour ce cas d'usage, mais on perd alors les avantages du JWT (claims embarqués, pas d'appel réseau).

---

## Conséquences

### Positives

- Validation du JWT sans appel réseau sur chaque requête (réduction de la latence)
- Claims (userId, rôle) directement accessibles dans le token — pas de lookup en base pour les autorisations courantes
- Interopérabilité naturelle avec les futurs microservices (chaque service valide le JWT indépendamment)
- Refresh token en base permet la révocation explicite si nécessaire

### Négatives

- Fenêtre de 15 minutes pendant laquelle un JWT volé reste valide même après révocation côté SSO
- La clé privée RS256 doit être gérée avec soin (rotation, stockage sécurisé) — complexité opérationnelle supplémentaire
- Le refresh token en base PostgreSQL introduit une écriture à chaque renouvellement de session

### Dette acceptée

La révocation immédiate d'un JWT n'est pas possible sans appel Redis supplémentaire sur chaque requête. Pour le contexte actuel (étudiants, risque faible), la fenêtre de 15 minutes est acceptable. Si le profil de risque évolue (paiements plus sensibles, profils organisateurs à révoquer d'urgence), il faudra envisager une blacklist Redis des JWT révoqués consultée à chaque requête — ce qui rapproche du modèle session.
